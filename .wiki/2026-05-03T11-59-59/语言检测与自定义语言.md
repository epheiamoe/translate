# 语言检测与自定义语言

## 自动语言检测：当源语言不确定时

用户将源语言选择为"自动检测"时，Moe Translate 不会盲目猜测——它会向 LLM 发送一次**非流式请求**，让模型判断文本语种并返回 ISO 639-1 语言代码。这一过程由两个函数协同完成。

### 1. 构建检测提示词

`buildDetectionPrompt(text)` 从 `prompts.yaml` 中读取 `detection_prompt` 模板，将用户输入的前 200 个字符填入 `{{text}}` 占位符。

```yaml
# prompts.yaml
detection_prompt: |
  Detect the language of this text and respond with only the ISO 639-1 language code:
  "{{text}}"
```

函数实现仅做模板替换：

```typescript
// src/lib/prompts/loadPrompts.ts
export function buildDetectionPrompt(text: string): string {
  const prompts = loadPrompts();
  return prompts.detection_prompt.replace('{{text}}', text);
}
```

[来源](src/lib/prompts/loadPrompts.ts#L317-L321)

### 2. 发送检测请求并解析结果

`detectLanguage()` 位于 `llmClient.ts`，它是一个标准的 **OpenAI-compatible 非流式 POST 请求**：

```typescript
export async function detectLanguage(
  apiBaseUrl: string,
  apiKey: string,
  prompt: string,
  model?: string
): Promise<string> {
  const url = `${apiBaseUrl}/chat/completions`;
  
  const response = await fetch(url, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${apiKey}`
    },
    body: JSON.stringify({
      model: model || 'deepseek-v4-flash',
      messages: [{ role: 'user', content: prompt }],
      stream: false  // 关键：非流式
    })
  });
  // ... 解析返回的 content，提取 ISO 639-1 代码
}
```

解析逻辑：从 LLM 的文本回复中用正则 `/^[a-z]{2}(-[A-Z]{2})?/` 提取最开头的两字母代码（如 `en`、`zh`、`ja`），若匹配失败则**默认返回 `'en'`**。

[来源](src/lib/llmClient.ts#L170-L207)

### 3. 在翻译流程中的调用

在 `useTranslation.ts` 的 `translate` 回调中，当 `sourceLang === 'auto'` 时触发检测：

```typescript
let resolvedSourceLang = sourceLang;

if (sourceLang === 'auto') {
  try {
    const detectionPrompt = buildDetectionPrompt(sourceText.slice(0, 200));
    resolvedSourceLang = await detectLanguage(
      config.baseUrl,
      config.apiKey,
      detectionPrompt
    );
  } catch {
    resolvedSourceLang = 'en';  // 异常降级
  }
}
```

检测结果 `resolvedSourceLang` 随后被传递给 `buildSystemPrompt` 和 `buildUserPrompt`，替换提示词模板中的 `{{source_lang}}` 占位符。

[来源](src/hooks/useTranslation.ts#L69-L84)

共 30 个条目（1 个自动检测 + 29 种具体语言）。`getSupportedLanguages()` 函数返回该映射的**浅拷贝**，确保外部不会意外修改缓存：

```typescript
export function getSupportedLanguages(): Record<string, string> {
  const prompts = loadPrompts();
  return { ...prompts.languages };
}
```

```typescript
export function getLanguageName(code: string): string {
  const prompts = loadPrompts();

}
```

## 自定义语言：扩展不受支持的语言

当用户需要翻译内置 29 种语言之外的语言（如梵语、世界语、古英语）时，可以通过设置面板添加自定义语言。

### 数据模型

`CustomLanguage` 接口定义了三种属性：

```typescript
// src/lib/db.ts
export interface CustomLanguage {
  id: string;
  name: string;
  promptSuffix: string;
}
```

- `id`：唯一标识符，格式为 `custom_${Date.now()}`，由前端自动生成
- `name`：用户输入的语言显示名称（如 "Sanskrit"）
- `promptSuffix`：附加到系统提示词中的指令后缀

[来源](src/lib/db.ts#L14-L18)

### 设置界面

在 `Settings.tsx` 中，用户通过两个输入框和一个"添加"按钮管理自定义语言：

```tsx
<section className="settings-section">
  <h3>{t('settings.customLanguages')}</h3>
  
  {/* 已有自定义语言的列表 */}
  {customLanguages.map(cl => (
    <div key={cl.id} className="custom-lang-item">
      <span>{cl.name}</span>
      <button onClick={() => handleRemoveCustomLang(cl.id)}>×</button>
    </div>
  ))}
  
  {/* 添加新语言的表单 */}
  <div className="add-custom-lang">
    <input placeholder="Language name (e.g., Sanskrit)" />
    <input placeholder="Prompt suffix for LLM" />
    <button onClick={handleAddCustomLang}>Add</button>
  </div>
</section>
```

`handleAddCustomLang` 创建新条目并更新本地状态：

```typescript
const handleAddCustomLang = () => {
  if (newLangName.trim() && newLangSuffix.trim()) {
    const id = `custom_${Date.now()}`;
    setCustomLanguages([
      ...customLanguages, 
      { id, name: newLangName, promptSuffix: newLangSuffix }
    ]);
    setNewLangName('');
    setNewLangSuffix('');
  }
};
```

保存设置时，`customLanguages` 数组随其他设置一并持久化到 IndexedDB。

[来源](src/components/Settings/Settings.tsx#L106-L113) | [来源](src/components/Settings/Settings.tsx#L439-L479)

### 注入语言选择器

`LanguageSelector` 组件在渲染下拉列表时，将自定义语言合并到选项集：

```typescript
// src/components/LanguageSelector/LanguageSelector.tsx
const allLanguages: Record<string, string> = {
  ...(showAutoOption ? { auto: t('languageSelector.autoDetect') } : {}),
  ...languages,  // 内置语言
  ...Object.fromEntries(
    settings.customLanguages.map(cl => [cl.id, cl.name])
  )  // 自定义语言
};
```

这意味着自定义语言直接出现在源语言/目标语言的下拉菜单中，用户可像选择内置语言一样选择它们。

[来源](src/components/LanguageSelector/LanguageSelector.tsx#L28-L35)

### 在提示词构建中的占位符替换

当用户选择了一个自定义语言（如 `custom_1712345678901`）作为源语言或目标语言时，提示词构建函数会经历如下流程：

```typescript
export function buildSystemPrompt(mode, sourceLang, targetLang, ...) {
  const prompts = loadPrompts();
  // 查找内置语言映射，找不到则使用代码本身
  const langName = prompts.languages[sourceLang] || sourceLang;
  const targetLangName = prompts.languages[targetLang] || targetLang;
  
  systemPrompt = systemPrompt
    .replace('{{source_lang}}', langName)    // → "custom_1712345678901"
    .replace('{{target_lang}}', targetLangName);
}
```

由于自定义语言的 ID（如 `custom_1712345678901`）不在 `prompts.languages` 中，回退逻辑 `|| sourceLang` 会直接使用该 ID 作为语言名称填充提示词。这就是当前代码中"替换占位符"的机制——它利用了 **fallback 到原始代码** 的行为。

[来源](src/lib/prompts/loadPrompts.ts#L126-L137) | [来源](src/lib/prompts/loadPrompts.ts#L217-L229)

> **注意**：`promptSuffix` 字段已存在于数据模型和 UI 中，但在当前版本的 prompt 构建管线中尚未被自动追加到系统提示词或用户提示词末尾。这是一个预留的设计锚点，未来版本可通过在 `buildSystemPrompt` 或 `buildUserPrompt` 中检测 sourceLang/targetLang 是否以 `custom_` 开头，然后追加对应的 `promptSuffix` 到提示词末尾，来实现对 LLM 的额外指令注入。
