# 构建工具链与配置

Moe Translate 的构建管线以 **Vite** 为核心，串联 TypeScript 编译、React JSX 转换、PWA 资源注入和测试环境配置。整个工具链的设计追求两个目标：**开发体验流畅**（HMR 热更新、类型检查即时反馈）和 **生产产物可靠**（先类型检查再打包、Service Worker 预缓存）。

## Vite 主配置：vite.config.ts

配置文件位于项目根目录，通过 `defineConfig` 导出完整的构建选项。[来源](vite.config.ts#L1-L63)

### @vitejs/plugin-react：JSX 编译

`react()` 插件启用 React 17+ 的 **自动 JSX 运行时**（`jsx-runtime`），开发者无需在每个组件文件顶部手动 `import React`。该插件还集成了 **Babel 语法转换**，支持 TypeScript 和最新的 JavaScript 特性。[来源](vite.config.ts#L5-L7)

### vite-plugin-pwa：PWA 清单与离线缓存

`VitePWA` 插件将 Vite 构建产物与 Service Worker 生成流程对接。配置分为三层：[来源](vite.config.ts#L8-L59)

1. **注册策略**：`registerType: 'autoUpdate'` 表示一旦检测到新版本 Service Worker，立即激活更新，用户无感知。[来源](vite.config.ts#L9)

2. **Web App Manifest**：定义了应用名称 `Moe Translate`、短名称 `Translate`、展示模式 `standalone`（全屏无浏览器 chrome）、启动方向 `portrait`。图标组包含三个条目：192px 常规图标、512px 常规图标、以及 `purpose: 'maskable'` 的 512px 自适应图标（适配 Android 自适应图标遮罩）。[来源](vite.config.ts#L11-L40)

3. **Workbox 运行时缓存**：`globPatterns` 指定预缓存所有静态资源（JS、CSS、HTML、字体、图片）。此外，通过 `runtimeCaching` 定义了两个外部字体 CDN 的缓存规则——Google Fonts（`fonts.googleapis.com`）和字体文件（`fonts.gstatic.com`）均采用 **CacheFirst** 策略，缓存有效期 365 天，最多保留 10 条记录。这意味着一旦字体被下载，后续访问完全离线可用。[来源](vite.config.ts#L42-L57)

> 关于离线缓存策略的详细分析，参见 [离线策略与性能优化](离线策略与性能优化.md)。

### build.rollupOptions：manualChunks

`manualChunks` 被显式设置为 `undefined`，这意味 Rollup 使用默认的分块策略——**不强制拆分 vendor 包**。Vite 内置的 `manualChunks` 默认逻辑（基于导入频率的自动拆分）生效，因此第三方依赖（如 React、Zustand）不会被统一打包为单个 `vendor` chunk，而是按实际引用关系自然分组。这一选择有助于减小初始加载体积，利用 HTTP/2 多路复用并行加载多个小 chunk。[来源](vite.config.ts#L60-L65)

## TypeScript 编译策略

项目采用 **双 tsconfig 结构**：一个面向应用源码，一个面向构建工具本身。

### tsconfig.json：应用编译配置

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "strict": true,
    "noEmit": true,
    "types": ["vite/client", "vitest/globals", "node"]
  },
  "include": ["src"]
}
```

关键设计决策：[来源](tsconfig.json#L1-L18)

- **`moduleResolution: "bundler"`**：这是 Vite 推荐的解析策略。与传统 Node.js 的 `node` 解析不同，bundler 模式允许省略文件扩展名、支持 `exports` 字段、允许直接导入 TypeScript 源文件（配合 `allowImportingTsExtensions`）。这是为 Vite（使用 esbuild 预构建依赖 + Rollup 打包）量身定制的配置。
- **`jsx: "react-jsx"`**：启用自动 JSX 运行时，与 `@vitejs/plugin-react` 协同工作。
- **`noEmit: true`**：TypeScript 仅作为类型检查器使用，实际代码编译由 Vite（esbuild）完成。
- **`types`** 数组包含了三个全局类型包：`vite/client`（提供 `import.meta.env` 等 Vite 特有类型）、`vitest/globals`（提供 `describe`、`it`、`expect` 等全局测试函数）、`node`（提供 Node.js 内置模块类型）。

### tsconfig.node.json：工具链编译配置

这是一个独立的 TypeScript 项目（`composite: true`），专门编译 `vite.config.ts` 等构建工具文件。[来源](tsconfig.node.json#L1-L13)

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "composite": true,
    "strict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true
  },
  "include": ["vite.config.ts"]
}
```

与主 tsconfig 的差异：
- `target: ES2022` 比主配置的 `ES2020` 更新，因为 Node.js 运行时对 ES2022 的支持已非常成熟。
- `noUnusedLocals` 和 `noUnusedParameters` 均设为 `true`，对工具代码执行更严格的未使用变量检查。
- `composite: true` 启用项目引用（project references），配合 `tsc -b`（build mode）实现增量编译。这也是 `npm run build` 中 `tsc -b` 命令能够高效工作的前提。

## Vitest 测试配置

测试环境通过独立的 `vitest.config.ts` 配置，而不是混入 `vite.config.ts`，保持关注点分离。[来源](vitest.config.ts#L1-L16)

```ts
export default defineConfig({
  plugins: [react()],
  test: {
    globals: true,
    environment: 'jsdom',
    setupFiles: ['./src/tests/setup.ts'],
    include: ['src/tests/**/*.test.{ts,tsx}'],
    coverage: {
      reporter: ['text', 'json', 'html'],
    },
  },
});
```

关键配置项：

- **`environment: 'jsdom'`**：使用 jsdom 模拟浏览器环境，使测试可以操作 DOM API（如 `document.querySelector`）。jsdom 24 是 devDependencies 中明确声明的依赖。
- **`setupFiles: ['./src/tests/setup.ts']`**：每个测试文件运行前自动执行 setup 脚本。该脚本（[来源](src/tests/setup.ts#L1-L34)）做了两件事：
  1. 全局导入 `@testing-library/jest-dom`（注入 `toBeInTheDocument()` 等自定义 DOM 匹配器）。
  2. 在 `window` 对象上 mock `indexedDB`，返回一个模拟的 IDB 实现，使不依赖真实 IndexedDB 的组件测试能够正常运行。

> 关于测试体系更完整的分析（包括 mock 策略和测试用例结构），参见 [测试体系：Vitest 与 Testing Library](测试体系：Vitest-与-Testing-Library.md)。

## npm Scripts 全景

从 `package.json` 中可以提取以下脚本：

| 命令 | 用途 | 底层命令 |
|---|---|---|
| `dev` | 启动 Vite 开发服务器，支持 HMR | `vite` |
| `build` | **先类型检查再构建**（双阶段） | `tsc -b && vite build` |
| `preview` | 本地预览构建产物 | `vite preview` |
| `test` | 以 watch 模式运行 Vitest 测试 | `vitest` |
| `test:run` | 单次运行所有测试（CI 适用） | `vitest run` |
| `test:ui` | 启动 Vitest UI 仪表盘 | `vitest --ui` |
| `lint` | ESLint 静态分析 | `eslint .` |
| `typecheck` | 单独运行类型检查（不生成产物） | `tsc --noEmit` |

[来源](package.json#L5-L18)

### `tsc -b && vite build`：执行顺序的意图

这是最值得深入理解的一个脚本。它由两个阶段组成：

1. **`tsc -b`**（TypeScript build mode）：利用 `composite` 项目引用机制执行增量类型检查，检查通过才继续。`-b` 标志比普通 `tsc` 更高效，因为它会缓存上次编译结果，只重新检查变更的文件。
2. **`vite build`**：类型检查通过后，由 Vite 调用 esbuild 进行转译（将 TS/JSX 转为普通 JS）和 Rollup 打包。

**为什么先类型检查再构建？** Vite 在开发模式下依赖 esbuild 进行转译，esbuild 速度极快但不执行类型检查——这是它比 `tsc` 快几个数量级的原因。因此，开发时类型错误不会阻碍 HMR 启动。但在生产构建中，必须确保输出代码没有类型错误，所以在 `vite build` 之前显式调用 `tsc -b`。这种 **开发宽松 + 生产严格** 的策略平衡了开发速度和代码可靠性。[来源](package.json#L7)

## 入口 HTML

`index.html` 是 Vite 应用的入口点，位于项目根目录而非 `public` 文件夹。它包含一个 `<script type="module" src="/src/main.tsx">` 标签，Vite 会自动解析此入口并递归收集所有依赖。[来源](index.html#L1-L16)

HTML 中声明的 meta 标签与 `vite-plugin-pwa` 生成的 manifest 协同工作：
- `theme-color: #ffffff` 与 manifest 中的 `theme_color` 一致。
- `apple-touch-icon` 指向 `/app-icon-192.png`，确保 iOS 添加到主屏幕时显示正确图标。
- `manifest` 链接指向 `/manifest.webmanifest`（VitePWA 插件自动生成的路径）。

## 构建产出结构

运行 `npm run build` 后，产物输出到 `dist/` 目录。核心产物包括：

- 经过哈希的文件名（如 `index.abc123.js`）以实现长效缓存
- 预缓存列表（由 Workbox 根据 `globPatterns` 自动生成）
- Service Worker 注册脚本
- 自动生成的 `manifest.webmanifest`
- 复制的 `public/` 目录静态资源（图标、`_headers`、`_routes.json`）

> 构建产物部署到 Cloudflare Pages 的具体配置，参见 [Cloudflare Pages 部署指南](Cloudflare-Pages-部署指南.md)。
