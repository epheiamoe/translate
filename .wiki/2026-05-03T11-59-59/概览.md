# 概览

Moe Translate 是一款以**离线优先**为核心理念的翻译 PWA 应用。它不依赖自己的后端服务器——所有翻译请求直接从你的浏览器发送到大语言模型（LLM）的 API。这意味着没有中间人截取数据，你的翻译内容只在你和 API 提供商之间流动。

## 三个核心价值

**离线优先**是 Moe Translate 的首要设计原则。借助 Service Worker 缓存策略，应用的界面、样式、脚本等静态资源在首次加载后即可离线运行。你可以在完全无网络的环境下浏览历史记录、管理设置，甚至编辑待翻译的文本。[来源](README.md#L4-L5)

**LLM 驱动**意味着它不依赖传统机器翻译引擎。每个翻译请求都是一次流式对话——应用向 LLM 发送精心构造的提示词（prompt），LLM 逐字返回翻译结果，你看到的是字符逐个出现的流式效果。这使得翻译风格、语气、术语都可以被精确控制。[来源](src/lib/llmClient.ts#L33-L45)

**PWA**（渐进式 Web 应用）让 Moe Translate 可以像原生应用一样安装在桌面或移动设备上，拥有独立的窗口和离线能力。更多细节请参阅 [PWA 安装与离线使用](pwa-安装与离线使用.md)。

## 一条数据从输入到输出的完整链路

理解 Moe Translate 最简单的方式是跟随一条数据请求走完全程。当用户在输入框中键入文本并点击翻译按钮时，以下流程在毫秒级时间内发生：

```mermaid
flowchart LR
    A[用户输入文本] --> B{语言检测?}
    B -->|auto| C[调用 LLM 检测语言]
    B -->|手动选择| D[使用选中语言]
    C --> D
    D --> E[从 prompts.yaml 构建提示词]
    E --> F[填充模板变量<br/>{{source_lang}}{{target_lang}}{{style}}]
    F --> G[streamTranslate 发送 SSE 请求]
    G --> H[逐块解析流式响应]
    H --> I{支持思考链?}
    I -->|是| J[提取 reasoning_content]
    I -->|否| K[提取 content 文本]
    J --> L[更新 ThinkingChain 组件]
    K --> M[逐字追加到 TranslationArea]
    L --> M
    M --> N[完整响应后保存到 IndexedDB]
    N --> O[更新历史面板和收藏]
```

这个流程的核心在于 **streamTranslate** 函数（定义在 `src/lib/llmClient.ts`）。它接收一个 `StreamTranslateOptions` 参数，包含 base URL、API key、模型 ID、system prompt 和 user prompt，然后通过 `fetch` 向 LLM 端点发送 SSE（Server-Sent Events）流式请求。每当收到一个数据块，就通过 `callbacks.onChunk` 回调推送给 UI。[来源](src/lib/llmClient.ts#L33-L45)

如果源语言设置为 "auto"，应用会先发送一次语言检测请求（复用同一个 LLM），返回 ISO 639-1 双字母代码后，再进入翻译流程。[来源](src/hooks/useTranslation.ts#L54-L64)

## 七家内置 LLM 提供商

Moe Translate 内置了七家 LLM 提供商的预配置信息，定义在 `src/lib/prompts/prompts.yaml` 的 `providers` 字段中。你可以直接在界面切换，无需手动填写 base URL。

| 提供商 | 标识键 | API 类型 | 默认端点 | 特色模型示例 |
|---------|--------|----------|----------|--------------|
| **DeepSeek** | `deepseek` | openai | `https://api.deepseek.com` | DeepSeek V4 Flash（支持思考链） |
| **OpenAI** | `openai` | openai | `https://api.openai.com/v1` | GPT-5.5 Flagship |
| **Anthropic** | `anthropic` | anthropic | `https://api.anthropic.com` | Claude Opus 4.7（支持思考链） |
| **Google Gemini** | `google` | google | `https://generativelanguage.googleapis.com/v1beta` | Gemini 3.1 Pro |
| **xAI Grok** | `xai` | openai | `https://api.x.ai/v1` | Grok 4.3（支持思考链） |
| **Mistral AI** | `mistral` | openai | `https://api.mistral.ai/v1` | Mistral Large |
| **Cohere** | `cohere` | openai | `https://api.cohere.com/v1` | Command A |

[来源](src/lib/prompts/prompts.yaml#L175-L381)

每个提供商下又包含多个模型，各自有不同的上下文窗口大小（max_context）、定价和是否支持思考链（reasoning）。关于如何配置 API Key 和切换模型，请参阅 [API 密钥与提供商配置](api-密钥与提供商配置.md)和 [ProviderSwitcher 与 ModelSwitcher 联动机制](providerswitcher-与-modelswitcher-联动机制.md)。

## 从项目中学到的四种架构模式

Moe Translate 虽然是一个"小应用"，但它采用的架构模式足以让你在 React 项目中举一反三。

### Zustand 持久化

全局状态使用 **Zustand** 管理，结合 `persist` 中间件实现状态的自动持久化。通过 `partialize` 配置精确控制哪些字段需要持久化（例如 `history` 和 `settings` 持久化，而 `targetText` 等临时状态不持久化），通过 `onRehydrateStorage` 在重新水合时进行数据迁移和兼容性处理。[来源](src/hooks/useAppStore.ts#L1-L10) 详见 [状态管理：Zustand 与持久化策略](状态管理-zustand-与持久化策略.md)。

### IndexedDB 封装

所有翻译历史、用户设置和 Token 用量统计都存储在浏览器的 IndexedDB 中。应用通过 `idb` 库（一个 IndexedDB 的 Promise 封装）操作名为 `translate-pwa-db` 的数据库，包含 `history`、`settings` 和 `tokenStats` 三个对象存储（object store）。[来源](src/lib/db.ts#L40-L44) 详见 [IndexedDB 数据层设计](indexeddb-数据层设计.md)。

### 流式 API 客户端

`streamTranslate` 函数是应用与 LLM 的通信枢纽。它处理 SSE 流的逐行解析、`AbortController` 的中止控制、`reasoning_content`（思考链内容）的分离提取，以及 Token 用量的追踪。不需要额外的 SDK，纯浏览器 `fetch` API 即可工作。[来源](src/lib/llmClient.ts#L33-L80) 详见 [LLM 流式 API 客户端架构](llm-流式-api-客户端架构.md)。

### YAML 驱动的提示词系统

所有与 LLM 对话的提示词模板都集中在 `src/lib/prompts/prompts.yaml` 中。这个文件定义了 system prompt（翻译模式、解析模式、文档翻译模式）、user prompt、样式描述、语言列表、提供商和模型配置。应用在运行时通过 `loadPrompts.ts` 将 YAML 解析为 TypeScript 对象，用 `{{source_lang}}` 等模板变量替换实际值，再发送给 LLM。[来源](src/lib/prompts/prompts.yaml#L1-L10) 详见 [YAML 驱动的提示词引擎](yaml-驱动的提示词引擎.md)。
